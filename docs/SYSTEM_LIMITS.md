# Tapnow 시스템 임계값 및 보안 분석 보고서

이 문서는 Tapnow 도장 관리 시스템이 사용하는 Firebase (Firestore, Auth) 환경의 한계 임계값(Threshold) 및 보안 적용 현황을 기록합니다.

## 1. Firestore 시스템 한계 임계값 (무료 Spark Plan 기준)

현재 프로젝트는 Firebase 무료 티어를 기반으로 구축되었습니다. 다음과 같은 임계값을 고려하여 설계 및 운영되어야 합니다.

### 데이터 읽기/쓰기 임계값
- **일일 쓰기 (Write)**: 20,000회 / 일
  - **분석**: 출석 체크(1회 쓰기), 신규 회원 등록(1회 쓰기), 요금제 수정 등. 소규모 도장(회원 1,000명 이하) 운영 시 하루 변동 사항이 2만 번을 넘는 경우는 거의 없어 **매우 안전**합니다.
- **일일 읽기 (Read)**: 50,000회 / 일
  - **분석**: **가장 주의해야 할 임계값**입니다. 상태 관리자(Zustand)가 초기 마운트 시 전체 회원, 출석 기록을 불러옵니다.
  - **위험 시나리오**: 도장 1곳의 회원이 1,000명일 때, 도장 관리자가 하루에 새로고침을 50번 하면 50,000 읽기가 발생하여 일일 한도에 도달합니다.
  - **향후 해결책**: 데이터가 누적(예: 과거 출석 기록)될 경우, `limit()` 및 `startAfter()`를 이용한 **페이지네이션** 또는 **최근 1달치 데이터만 기본 로딩**하는 구조로 로직을 최적화해야 합니다.

### 스토리지 및 네트워크
- **저장 공간**: 1GB (Firestore)
  - **분석**: 텍스트 데이터의 용량은 매우 적으므로 수년간 사용해도 한계치 도달 위험은 낮습니다. 프로필 이미지 저장은 Firebase Storage(기본 5GB)를 활용합니다.
- **네트워크 대역폭**: 10GB / 월
  - 텍스트 위주 시스템이므로 10GB는 월 단위 서비스 유지에 충분한 양입니다.

## 2. 보안 적용 현황 및 가이드

Tapnow는 클라이언트(브라우저)에서 직접 Firestore에 접근하므로, NoSQL 보안 규칙(`firestore.rules`)을 통한 접근 제어가 필수입니다.

### 적용된 보안 원칙
1. **인증 강제화 (Authentication Required)**: 로그인하지 않은 사용자(`request.auth == null`)는 모든 데이터 쓰기/읽기 접근이 원천 차단됩니다.
2. **데이터 격리 (Multi-tenancy Isolation)**: 모든 문서(`members`, `plans`, `attendances`, `payments`)에는 `gymId` 필드가 저장됩니다. Firebase Rules에서 `gymId`와 현재 로그인한 사용자의 `uid`가 일치하는지(`resource.data.gymId == request.auth.uid`) 강제 검증하여, **타 도장의 데이터 열람 및 변조를 방지**합니다.
3. **슈퍼 관리자 권한**: 지정된 본사 관리자(`hjbyun0921@naver.com`)만 모든 데이터에 읽기/쓰기 권한을 가지도록 `isSuperAdmin()` 룰을 구현했습니다.

### 보안 규칙 적용 방법
방금 프로젝트 루트에 생성된 `firestore.rules` 파일을 실제 Firebase 서버에 적용해야 합니다.
Firebase 콘솔(웹)에서 적용하려면:
1. [Firebase Console] -> [Firestore Database] -> [Rules] 탭으로 이동합니다.
2. 에디터 내용을 삭제하고 로컬의 `firestore.rules` 내용을 복사하여 붙여넣습니다.
3. **[Publish (게시)]** 버튼을 눌러 적용합니다.

## 결론 및 권장 사항

1. 현재의 데이터 `cleanData` (NaN, undefined 필터링)와 Firestore Rules를 통해 **프론트엔드-백엔드 양방향 보안 구멍을 모두 차단**하였습니다.
2. 당분간은 무료 티어로 충분히 서비스 런칭이 가능하나, 데이터베이스의 '한 번의 읽기'당 문서 1개당 비용이 책정되므로 런칭 후 `1. 전체 데이터 다운로드`를 지양하고 **2. 기간별 데이터 페칭(1주일 치 출석 조회 등)**으로 전환하실 것을 권장합니다.
